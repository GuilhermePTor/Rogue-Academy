MODULE_04 // WEEKS 6–7

Model Deployment

Online, batch, and streaming — three deployment modes — ~10–12 hrs
Deployment is not "put it in Flask and call it done." There are three fundamentally different patterns: online serving (real-time, low latency), batch scoring (scheduled, high throughput), and streaming (event-driven, near-real-time). Each has different cost, latency, and operational profiles. The Zoomcamp teaches all three using the taxi model. After this module, you'll be able to choose the right pattern for any use case and defend your choice in a system design interview.

MODULE_05 // WEEK 8

Model Monitoring

Data drift, concept drift, Evidently, Grafana, Prometheus — ~8–10 hrs
Deploying a model is not the end — it's the beginning of a different problem. Models degrade silently. Your input data distribution drifts. User behaviour changes. New categories appear that your model has never seen. Without monitoring, you find out the model is broken when someone complains. With monitoring, you find out three days before anyone notices. The Zoomcamp covers Evidently for drift detection and Grafana + Prometheus for dashboards.

[M5.1]Evidently — Data Quality & Drift Reports ZOOMCAMPGITHUB ↗
// LEARN
CONCEPTReference vs current dataset: Evidently compares your current production batch against a reference dataset (the training distribution). The distance between them is your drift signal
CONCEPTDrift detection tests: Jensen-Shannon divergence for distributions, Wasserstein distance for continuous features, chi-squared for categorical. Know which applies to which feature type
CONCEPTEvidently Report vs Test Suite: Reports are human-readable HTML; Test Suites return pass/fail for automation. Use Test Suites in pipelines, Reports for debugging
CONCEPTData quality metrics: missing values, value range violations, schema changes. Run these before drift detection — a schema change is often more urgent than statistical drift

// LEARN
THREATLLM01: Prompt Injection — attacker injects instructions into user input or retrieved context to override system prompt behaviour. The most exploited LLM vulnerability in production
THREATLLM02: Insecure Output Handling — LLM output rendered as HTML or executed as code without sanitisation. Leads to XSS, SSRF, and RCE in agent systems
